# GUI Testing — ExpenseApp (Semester Project Task 4)

## Overview
This document describes the GUI testing strategy, tooling, and artifacts for the ExpenseApp. Tests target the following modules:
- Budget
- Category
- Expense
- BudgetAI service

This repository contains:
- Livewire component tests (PHPUnit/Pest style) — fast, deterministic unit tests.
- Playwright E2E tests using Page Object Model (POM) — cross-browser, parallel execution.

## Tools Used
- Playwright — chosen for robust cross-browser automation, auto-waiting, network interception, and built-in parallelization. Preferred over Selenium for reliability and modern API.
- Livewire component testing (via PHPUnit/Pest) — for unit/component-level assertions without launching browsers.

Why Playwright over Selenium:
- Built-in cross-browser engine control (Chromium, Firefox, WebKit) without external drivers.
- Better stability via auto-waiting, tracing, and easier CI integration.
- Codegen/recorder (`npx playwright codegen`) provides the recorder-style workflow required by the assignment.

## Test Structure
- `tests/Feature/Livewire/*` — component tests for Livewire components.
- `tests/Unit/*` — unit tests for models and services.
- `tests/e2e/pages/*` — Playwright Page Object Model classes.
- `tests/e2e/specs/*` — end-to-end spec files using POM.
- `playwright.config.ts` — cross-browser and parallel config.

## Playwright Parallel Cross-Browser Execution
Example config (already present): `playwright.config.ts` runs tests in parallel workers and defines `chromium` and `firefox` projects. Run locally:

```bash
npm install -D @playwright/test
npx playwright install
npx playwright test
```

CI note: ensure `npx playwright install` is run in the pipeline before `npx playwright test`.

## Recorder-Style Workflow (equivalent)
Use Playwright Codegen to record a user flow and generate a test script:

```bash
npx playwright codegen http://localhost:8000/expenses/create
```

The generated TypeScript snippet will include page interactions. Example generated snippet for "Add Expense":

```ts
// Generated by `npx playwright codegen`
await page.goto('http://localhost:8000/expenses/create');
await page.fill('input[name="title"]', 'Recorded Lunch');
await page.fill('input[name="amount"]', '7.50');
await page.fill('input[name="date"]', '2026-06-12');
await page.click('button:has-text("Save")');
await page.waitForSelector('text=Expense created successfully');
```

This satisfies the "record and playback" requirement: the team can record flows, commit the generated test, and evolve it into a maintainable POM-based test.

## Test Scenarios Covered (high level)
- Component tests: validation and persistence for `ExpenseForm`, budget calculations for `Budget` model, totals in `BudgetList`, and `BudgetAIService` historical data check.
- E2E tests: navigation to dashboard, category creation, expense creation, and requesting BudgetAI recommendation.

## Challenges & Solutions
- Livewire dynamic updates: use component tests for state and Playwright for E2E; Playwright's auto-wait handles many Livewire async requests.
- Auth in E2E: prefer test users or API-auth endpoints to avoid brittle UI login in CI. Alternatively seed a known user and use login steps in tests.
- Parallelism flakiness: isolate data per-test (use unique names) and run database in transaction or use separate containers/database instances for workers.
